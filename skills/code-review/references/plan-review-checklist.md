# Plan Review Checklist

## Architecture

- [ ] 데이터의 source of truth가 명확하다.
- [ ] 데이터 흐름이 입력부터 출력까지 기술되어 있다.
- [ ] 공유 계약(contracts)이 이름과 위치와 함께 정의되어 있다.
- [ ] 기존 아키텍처를 우회하지 않고 확장한다.
- [ ] 새로운 추상화에 정당한 근거가 있다.

## Parallel Safety

- [ ] 공유 파일에 소유자가 지정되어 있다.
- [ ] Foundation 작업이 먼저 반영된다.
- [ ] 각 워크스트림에 비소유 경계가 정의되어 있다.
- [ ] 머지 순서가 결정적(deterministic)이다.

## Edge Cases

- [ ] 빈 입력 / 설정 누락
- [ ] 잘못된 입력 / 스키마 에러
- [ ] 중단(abort)/취소(cancel) 경로
- [ ] 권한 거부 / 사용자 거절
- [ ] 부분 실패 / 재시도 / 타임아웃
- [ ] 재개(resume)/재시작(restart) 동작 (해당 시)

## Security & Safety

- [ ] 시크릿이 로그에 남지 않는다.
- [ ] 외부 네트워크 호출이 제한되거나 회피 가능하다.
- [ ] 파괴적 작업이 권한 게이트 뒤에 있다.
- [ ] 파일 쓰기가 범위 지정되어 있고 복구 가능하다.

## Tests

- [ ] 순수 로직에 대한 유닛 테스트
- [ ] 연결에 대한 통합 테스트
- [ ] 이전에 취약했던 동작에 대한 회귀 테스트
- [ ] 실제 사용자 흐름에 대한 smoke 테스트
- [ ] 전체 typecheck/build/test가 최종 게이트로 나열되어 있다

## Verdict Labels

- `ready`: 계획을 지금 구현할 수 있다.
- `needs revision`: 대부분 맞지만 수정이 필요하다.
- `blocked`: 정보가 부족하거나 아키텍처가 안전하지 않다.
